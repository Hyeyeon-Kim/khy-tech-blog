# Kezzle · 유사 케이크 검색 — 결과는 10건인데 왜 검색 비용은 전체 데이터에 비례했을까?

> MongoDB 2dsphere와 FAISS 검색 경로를 재설계한 과정

## Kezzle은 어떤 서비스인가?

**Kezzle은 사용자가 원하는 케이크 디자인을 내 주변 케이크 매장에서 찾아주는 위치 기반 디자인 탐색 서비스입니다.**

기존 주문과정은 번거로웠습니다.
사용자는 인스타그램에서 케이크 키워드를 검색하고, 피드를 훑으며 마음에 드는 디자인을 찾은 뒤, 매장 프로필에 들어가 위치를 확인해야 했습니다. 매장이 멀면 다시 검색으로 돌아가야 했고, 원하는 디자인과 가까운 매장을 찾을 때까지 이 과정을 반복해야 했습니다.

이 과정을 **이미지 유사도 기반 검색**으로 줄이고자 했습니다.
사용자가 원하는 케이크 이미지를 선택하면, 비슷한 디자인을 제공하는 주변 매장을 바로 탐색할 수 있도록 했습니다.

![유사 케이크 검색 UI]({{ '/assets/similar-cake-ui.svg' | relative_url }})

하지만 운영과정에서 입점 매장이 4.2배 성장(120→503) 동안 검색이 **선형으로 느려지는** 문제가 발생했습니다.
![입점 매장 증가에 따른 검색 응답 시간 변화]({{ '/assets/search-latency-chart.png' | relative_url }})

유사 케이크 검색은 주문 전환의 시작점이기 때문에 응답 지연은 곧 탐색 이탈로 이어져, 지연이 발생하던 구간에서 주문 버튼 활성률은 **16%에서 7% 미만**으로 하락했습니다.

따라서 이탈을 줄이기 위해 검색 품질은 유지하면서 응답 속도와 가용성을 확보할 수 있는 구조로 검색 시스템을 개선했습니다.

## 데이터가 늘수록 검색이 느려졌다

![기존 검색 아키텍처]({{ '/assets/architecture-before.png' | relative_url }})

기존 시스템에서는 API가 최종 10건을 반환하더라도, 검색 과정에서는 전체 케이크를 대상으로 유사도를 먼저 계산한 뒤 위치 필터링을 적용했습니다.

| 케이크 데이터 수 | 최종 반환 수 | 기존 검색 P95 |
| ---------------- | ------------ | ------------- |
| 10.2만 (120매장) | 10건         | 132ms         |
| 26.8만 (315매장) | 10건         | 713ms         |
| 42.7만 (503매장) | 10건         | 1,680ms       |

반환량은 동일했지만 응답 시간은 데이터 수에 비례해 늘었습니다. 즉, 병목은 최종 반환량이 아니라 위치 필터링 전에 처리되는 전체 데이터량에 있었습니다.

검색 이후 단계에서도 전체 데이터의 영향을 받는지 확인하기 위해, 유사도 계산(FAISS 검색)과 위치·매장 정보 결합(cake 결합) 단계로 나누어 측정했습니다.

| 전체 케이크 수 | FAISS 검색 P95 | cake 결합 P95 | 전체 응답 P95 |
| -------------: | -------------: | ------------: | ------------: |
|         10.2만 |         49.8ms |        45.4ms |         132ms |
|         26.8만 |          339ms |         165ms |         713ms |
|         42.7만 |          935ms |         410ms |       1,680ms |

FAISS 검색은 전체 케이크 벡터를 대상으로 거리 계산을 수행해 데이터 수에 비례했고, cake 결합 또한 FAISS가 생성한 전체 점수 배열을 전달받아 처리했기 때문에, 최종 반환 수와 무관하게 전체 케이크 수 N의 영향을 받았습니다. 실제 503매장 규모에서는 응답이 **약 1.7초**까지 늘어났고, 점수 배열을 통째로 전달하던 결합 단계는 데이터가 더 커지면 그 자체로 한계에 부딪히는 구조였습니다.

문제가 위치 조건의 유무가 아니라 적용 시점에 있었기에, 개선 방향은 검색 전에 위치 조건으로 후보를 먼저 줄이는 것으로 정했습니다.

## 위치 후보 기반으로 검색 경로를 재설계

![위치 후보 기반으로 재설계한 v2 아키텍처]({{ '/assets/architecture-v2.png' | relative_url }})

먼저 `stores.location`에 `2dsphere` 인덱스를 걸어, 전체 매장을 확인하는 대신 반경 내 매장만 셀 범위 스캔으로 조회하고 케이크 검색에 사용할 후보 ID를 추출했습니다.

이 후보 ID를 `FAISS IDSelectorBatch`에 전달해 고비용 거리 계산을 전체 벡터가 아닌 후보 벡터로 제한했습니다. 또한 전체 크기의 점수 배열을 MongoDB에 전달하던 결합 과정을 제거하고, 후보 점수만 애플리케이션에서 처리했습니다.

| 전체 케이크 수   | 기존 P95 | 개선 P95 |
| ---------------- | -------- | -------- |
| 10.2만 (120매장) | 132ms    | 54ms     |
| 26.8만 (315매장) | 713ms    | 66ms     |
| 42.7만 (503매장) | 1,680ms  | 164ms    |

실제 카탈로그 규모(503매장 ≈ 42.7만 디자인) 기준 P95는 1,680ms에서 164ms로 약 **10배** 단축됐습니다(데이터가 클수록 격차가 벌어져 2.4 → 10.8배). FAISS 거리 계산은 935ms → 44ms, 점수 결합은 410ms → 2ms로 줄었습니다.

### 후보만 넘긴 방식도 사실 O(N)

개선 경로는 큰 폭으로 빨라졌지만 전체 검색 복잡도가 O(후보)로 바뀌었다고 단정할 수는 없었습니다.

`FAISS IndexFlatL2`는 `IDSelectorBatch`를 써도 전체 벡터 ID를 순회하며 후보 포함 여부를 확인합니다. 고비용 거리 계산은 후보로 제한되지만, 후보인지 판별하는 비용은 여전히 전체 데이터 N에 비례합니다. 후보를 100건으로 고정하고 전체 데이터만 늘려 다시 측정했습니다.

| 전체 벡터 수 | 부분 검색 P95 |
| -----------: | ------------: |
|       80,000 |        0.37ms |
|      320,000 |        1.56ms |

현재 규모에서는 상수가 작아 평탄해 보였지만 데이터가 4배 늘자 검색 시간도 약 4배 증가했습니다.

### 그럼 벡터를 비슷한 것끼리 나눠둔다면?

매번 전체 벡터를 다 보지 않으려면 벡터를 유사도로 군집화해 cluster에 나눠두고 질의와 가까운 몇 개의 방만 열어보는 방법이 있습니다.

대신 trade-off로 진짜 가장 비슷한 케이크가 열어보지 않은 cluster에 들어 있을 수 있습니다.

| 검색 방식               | recall@10 | 검색 P95 |
| ----------------------- | :-------: | -------: |
| 위치 후보 한정 정확검색 |    1.0    |   0.09ms |
| 전역 정확검색           |    1.0    |    5.9ms |
| IVF (nprobe=64)         |   0.95    |    5.6ms |

실제 ViT 임베딩 25,000건으로 측정했을 때 nprobe을 늘리면 recall은 올라가지만 그만큼 느려져서 recall 0.95를 맞추면 5.6ms로 전수검색(5.9ms)과 거의 같아졌습니다.

### 검색 기준이 위치라면 인덱스도 위치로 나눠야 한다

IVF는 벡터를 이미지 유사도 기준으로 나눕니다. 비슷한 이미지끼리 같은 방에 넣고, 질의와 가까운 방만 열어보는 방식입니다.

그런데 Kezzle의 경우 사용자가 원하는 것은 전역에서 가장 비슷한 케이크가 아니라, 내 주변에 있는 비슷한 케이크입니다. 그래서 인덱스도 이미지 유사도가 아니라 지역 기준으로 나누는 편이 더 나은 방향이라고 판단했습니다.

지역별로 벡터 인덱스를 두면, 검색할 때는 현재 위치의 반경에 걸치는 샤드만 보면 됩니다. 다른 지역은 처음부터 검색하지 않습니다. 덕분에 전체 데이터가 늘어나도 매번 전부를 훑을 필요가 없고, 샤드 안에서는 전수검색을 하기 때문에 recall 손실도 없습니다.

| 전체 케이크 수 | 전역 검색 |     지역 샤드(정확) |
| -------------: | --------: | ------------------: |
|         25,000 |    22.2ms | 0.28ms (recall 1.0) |
|        100,000 |    84.5ms | 0.30ms (recall 1.0) |

물론 지역으로 나누면 셀 경계 문제가 생깁니다. 한 셀만 보면 바로 옆 셀에 있는 후보를 놓칠 수 있습니다. 실제로 한 셀만 검색했을 때는 recall이 0.55까지 떨어졌습니다.

하지만 이 문제는 이웃 셀까지 같이 보면 해결됩니다. 3×3 셀을 함께 검색하자 recall은 다시 1.0까지 올라왔습니다. IVF에서 생기는 recall 손실은 가까운 클러스터를 제대로 고르지 못하면 피하기 어렵지만, 지역 샤딩의 경계 문제는 이웃 셀을 얼마나 더 볼지로 조절할 수 있습니다.

### 무엇을 근처로 볼 것인가

마지막으로 하나 더 짚고 넘어갈 부분이 있습니다. 실제 매장은 고르게 퍼져 있지 않습니다. 도심(홍대, 강남, 성수)에는 몰려 있고, 외곽이나 지방으로 갈수록 듬성듬성해집니다. 트래픽도 대체로 사람이 많은 곳에 몰립니다.

여기서 처음 세웠던 가설과 실제 구현 사이에 차이가 생겼습니다. 처음 가설은 “반경 3km 안에서 찾는다”였지만, 실제 구현은 “가장 가까운 N개 매장을 먼저 가져온다”에 가까웠습니다.

| 방식                       | 후보 수 p99 | 지연 p99 | 희소지역 문제                   |
| -------------------------- | ----------: | -------: | ------------------------------- |
| 반경(3km 가설과 일치)      |     3,500건 |   3.16ms | 빈 결과 발생                    |
| 가장 가까운 N개(실제 구현) |       100건 |   0.13ms | "근처"가 반경 1.5→2.52로 멀어짐 |

반경 방식은 “근처”의 의미가 명확합니다. 3km 안이면 근처고, 아니면 아닙니다. 다만 도심처럼 매장이 많은 지역에서는 후보가 크게 늘어 지연 시간이 길어지고, 매장이 적은 지역에서는 결과가 비어버릴 수 있습니다.

반대로 가장 가까운 N개 방식은 지연이 안정적입니다. 후보 수가 항상 일정하기 때문입니다. 대신 희소 지역에서는 N개를 채우기 위해 검색 범위가 계속 넓어집니다. 이 경우 사용자가 보기에는 더 이상 “근처”라고 하기 애매한 매장까지 포함될 수 있습니다.

결국 실제 구현은 nearest-N을 기본으로 두되, 최대 반경 3km를 함께 걸었습니다. 도심에서는 가까운 매장만 빠르게 가져오고, 희소 지역에서는 N개를 다 채우지 못하더라도 3km 밖의 매장은 제외했습니다.

빈 결과가 나올 수는 있습니다. 하지만 “근처”라고 보기 어려운 매장을 보여주는 것보다는, 결과가 비더라도 기준을 지키는 쪽이 맞다고 판단했습니다.

## 정리하며

입점 매장이 4.2배 늘면서 기존 검색은 데이터 증가에 따라 그대로 느려졌습니다. 이를 해결하기 위해 검색의 출발점을 전역 이미지 유사도에서 위치 후보로 옮겼습니다.

- 전체 유사도 검색 → 반경 후보 한정 검색으로 P95 1,680ms → 164ms (약 10배 단축)
- 검색 기준(위치)에 맞춰 인덱스를 지역으로 분할, 이웃 셀(3×3)까지 검색해 recall 1.0 유지
- 마지막으로 nearest-N에 최대 반경을 더해, 성능과 “근처”의 의미 사이의 균형을 맞췄습니다.
